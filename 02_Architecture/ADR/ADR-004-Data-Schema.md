# 1. Context

CCOP integrates multiple edge platforms, device types, and sensor classes operating under real-world field conditions.

Without a strict telemetry schema standard, the platform will fail through:

- inconsistent producer payloads
- consumer incompatibility
- schema drift across firmware versions
- broken processing pipelines
- incorrect dashboards and analytics
- weak traceability during incident investigation

A unified, versioned, enforceable schema standard is required for all production telemetry.

---

# 2. Decision

All CCOP telemetry shall follow a standardized, versioned, controlled message schema.

This standard applies to:

- edge devices
- ingestion services
- Kafka producers
- stream processors
- downstream consumers
- storage adapters
- analytics and dashboard services

No uncontrolled, ad hoc, or unversioned production payloads are permitted.

---

# 3. Schema Model

Every telemetry message shall use a two-level structure:

1. **Envelope**  
   Mandatory, fixed, versioned, and common across all telemetry types

2. **Payload**  
   Controlled, telemetry-specific content for air quality, weather, node status, and future approved event classes

The envelope must remain stable across telemetry types.

---

# 4. Mandatory Envelope Definition

All telemetry messages shall contain the following fields.

| Field | Type | Required | Description |
|---|---|---:|---|
| schema_version | string | Yes | Schema version, for example `1.0` |
| message_id | string | Yes | Globally unique identifier for the event |
| event_type | string | Yes | Approved event class |
| device_id | string | Yes | Unique device identifier |
| node_type | string | Yes | Platform type such as `TerraNode`, `OnMove`, `SeaEye`, `SkyNode` |
| event_time_utc | string | Yes | Event time generated by the device in ISO8601 UTC |
| ingest_time_utc | string | Yes | Time assigned by ingestion layer in ISO8601 UTC |
| gps | object | Yes | Geographic coordinates object |
| gps_status | string | Yes | GPS quality status |
| firmware_version | string | Yes | Device firmware version |
| sensor_status | object | Yes | Sensor health and quality indicators |
| payload | object | Yes | Telemetry-specific content |

---

# 5. Approved Event Types

The following event types are initially approved:

- `air_quality`
- `weather`
- `node_status`

Additional event types require formal review and approval.

No free-form event types are permitted in production.

---

# 6. GPS Structure Standard

The `gps` field shall use the following structure:

```json
{
  "lat": 25.2854,
  "lon": 51.5310
}
GPS Rules
gps must always exist in the message
if location is unknown or invalid, lat and lon may be null
gps_status must always describe the quality state
fake coordinates, default coordinates, or placeholder coordinates are prohibited
Approved GPS Status Values
valid
invalid
stale
Meaning
valid = current trusted fix
invalid = no usable fix
stale = last known position carried forward beyond acceptable freshness threshold
7. Timestamp Standard

All timestamps must:

be in UTC
use ISO8601 format
include date and time
include trailing Z
Required Timestamps
event_time_utc

Time assigned by the device when the measurement or event occurred.

ingest_time_utc

Time assigned by the ingestion layer when the event entered CCOP backend control.

Rules
both timestamps are mandatory
downstream systems must not overwrite event_time_utc
ingest_time_utc must be assigned or preserved by the ingestion path
consumer logic must not assume these timestamps are identical
8. Message Identity Standard

message_id must be unique per event.

Rules
must support replay-safe traceability
must remain unchanged end-to-end
must not be regenerated by downstream consumers
must be suitable for duplicate detection and audit tracing

UUID-style identifiers are recommended.

9. Device Identity Standard

device_id must uniquely identify the producing device.

Rules
no shared device IDs across multiple physical systems
no ambiguous renaming without controlled migration
device IDs must remain stable across restarts
analytics and routing must treat device_id as the primary producer identity
10. Node Type Standard

node_type identifies the platform class.

Examples:

TerraNode
OnMove
SeaEye
SkyNode
Rules
values must be controlled
spelling and capitalization must remain consistent
no ad hoc node type names in production
11. Sensor Status Standard

sensor_status is mandatory and shall report quality and health state for relevant sensors.

Example:

{
  "pm25": "ok",
  "no2": "ok",
  "o2": "calibration_due",
  "gps": "stale"
}
Purpose

This field is required to:

expose sensor health
indicate calibration and trustworthiness
distinguish valid absence from sensor failure
support operations and maintenance workflows
Approved Status Examples
ok
warning
failed
offline
calibration_due
invalid

Additional controlled values may be introduced through versioned updates.

12. Payload Rules

The payload section contains telemetry-specific measurements.

Mandatory Rules
payload must be an object
field names must be stable and versioned
no mixed naming conventions
units must be explicit and stable
null values must be explicit where data is unavailable
field meanings must not silently change across versions
Prohibited
ambiguous abbreviations without definition
mixed units across equivalent devices
changing field meaning without schema version increment
packing unrelated event classes into the same payload format
13. Naming Convention Standard

Field names shall follow these rules:

lowercase only
underscore-separated
no spaces
no special formatting
no vague abbreviations unless standardized
Good Examples
temperature_c
pressure_hpa
wind_speed_mps
battery_voltage_v
Bad Examples
Temp
pressure
WindSpeed
data1

Where units matter, units shall be encoded in field names or fully defined in schema documentation.

14. Units Standard

Units must be consistent across the platform.

Examples:

temperature in c
pressure in hpa
wind speed in mps
particulate concentration in ug_m3
Rules
one field name must map to one unit only
the same measurement type may not switch units by device or firmware version
unit changes require schema review and version change
15. Null and Missing Value Rules

Missing data must be explicit.

Rules
unknown values must be represented as null
invalid values must not be silently omitted
producers must not remove mandatory envelope fields
consumers must distinguish between null, invalid, and absent-by-design cases

Silent omission is prohibited for mandatory fields.

19. Schema Versioning Rules

schema_version is mandatory in every message.

Rules
non-breaking additions may increment minor version
breaking changes require version change
old and new versions may need transitional support during migration
field meaning changes without version increment are prohibited

Producers and consumers must not assume versionless compatibility.

20. Validation Requirements

The ingestion layer or schema validation stage must validate:

mandatory envelope fields
allowed event types
timestamp format
required object types
approved status values where defined
payload object presence

Messages that fail validation must not be silently treated as valid production telemetry.

21. Consumer Requirements

All downstream consumers must:

parse envelope fields first
respect schema_version
preserve message_id
handle replay and duplicates correctly where required
tolerate null values in controlled ways

Consumers must not depend on undocumented payload variants.

22. Data Quality Rules

The following are mandatory:

invalid values must be flagged, not hidden
sensor health must always be represented in sensor_status
stale GPS must be marked explicitly
no silent field dropping for mandatory envelope fields
no fabricated replacement values for missing production data
23. Compliance

This schema standard is mandatory for:

edge devices
ingestion layer
Kafka producers
stream processors
storage adapters
dashboards and analytics consumers

Any deviation requires formal review and ADR approval.

24. Consequences
Positive
consistent telemetry across the platform
lower integration risk
easier validation and debugging
safer schema evolution
more reliable analytics and operational dashboards
Negative
requires discipline from firmware and backend teams
schema changes must be governed
validation must be maintained as part of operations
